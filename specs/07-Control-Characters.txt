1. OVERVIEW
================================================================================

The 32 ASCII control characters (0x00-0x1F) serve as the command vocabulary
of the CBDF format. They appear in two contexts:

  A. STRUCTURAL: Separating major sections and sub-sections of the document.
     These appear at the document level (between sections) and within the
     Styles section (between sub-tables).

  B. INLINE: Appearing within the Text section to switch styles, insert
     elements, and mark semantic boundaries. These are interspersed with
     printable UTF-8 text.

PLAIN-TEXT EXTRACTION RULE: A plain-text reader strips ALL bytes 0x00-0x1F
from the text section EXCEPT:
  - 0x09 (HT) = Horizontal Tab
  - 0x0A (LF) = Line Feed
  - 0x0D (CR) = Carriage Return
The remaining printable UTF-8 bytes are the readable text.

3. DETAILED COMMAND SPECIFICATIONS
================================================================================

--- 0x00 NOP (NUL): No Operation / Padding ---

  Purpose: Ignored by the parser. Used for byte alignment in style records
  or as filler in fixed-size fields.
  Payload: None.
  Context: May appear anywhere. Always ignored.


--- 0x01 SUBJECT_START (SOH): Styled Subject ---

  Purpose: Marks the beginning of the styled subject text within the Text
  section. Only meaningful when the document type is "email" (set in meta).
  The subject text between SOH and the next ETX is the styled version of the
  email subject. The plain subject in the meta section is used for inbox
  listings; the styled subject is rendered when the email is opened.

  Payload: None. Styled text follows until the next DC4 (0x14) or ETB (0x17).
  Context: Text section only. Must appear INSIDE the STX/ETX block (after
  STX, before other body content). This keeps the parser in a single mode --
  it does not need to handle styled content before STX.
  Usage:
    [STX] [SOH] [DC1 style_index] Subject text here [DC4] Body text... [ETX]

  Plain-text extraction: SOH is stripped. The subject text remains as part
  of the plain text output.


--- 0x02 TEXT_START (STX): Start of Text ---

  Purpose: Marks the beginning of the document body text. Mandatory in every
  document that has a Text section.
  Payload: None.
  Context: Text section. Appears once.


--- 0x03 TEXT_END (ETX): End of Text ---

  Purpose: Marks the end of the document body text. Since each section has
  a 4-byte length prefix, ETX is technically redundant -- the length prefix
  is authoritative for determining section boundaries. ETX serves as a
  VALIDATION CHECK: the parser can verify that ETX appears at the position
  indicated by the length prefix. If it doesn't, the document may be corrupt.
  Encoders SHOULD emit ETX. Parsers SHOULD use the length prefix as the
  primary boundary and MAY validate ETX if present.
  Payload: None.
  Context: Text section.


--- 0x04 DOC_END (EOT): End of Document ---

  Purpose: Explicit end-of-file marker. Optional -- the parser may also use
  the section length prefix or actual EOF to determine the end.
  Payload: None.
  Context: After the last section.


--- 0x05 RESERVED_05 (ENQ): Reserved (Phase III) ---

  Possible future use: Interactive resource request from server.
  NOT functional in Phase II.


--- 0x06 RESERVED_06 (ACK): Reserved (Phase III) ---

  Possible future use: Interactive acknowledgment / form response.
  NOT functional in Phase II.


--- 0x07 RESERVED_ALERT (BEL): Reserved (Phase III) ---

  Possible future use: Client-side alert or notification sound.
  NOT functional in Phase II.


--- 0x08 RESERVED_BKSP (BS): Reserved (Phase III) ---

  Possible future use: Backspace for typing/animation effects in Logic.
  NOT functional in Phase II.


--- 0x09 TAB (HT): Horizontal Tab ---

  Purpose: Standard tab character. PRESERVED in plain-text extraction.
  Payload: None.
  Context: Text section.


--- 0x0A LINE_BREAK (LF): Line Feed ---

  Purpose: Standard Unix line break. PRESERVED in plain-text extraction.
  Payload: None.
  Context: Text section.


--- 0x0B PARA_BREAK (VT): Paragraph Break ---

  Purpose: Marks a semantic paragraph boundary. Renderers should insert
  vertical spacing (larger than a line break). Unlike LF which is a simple
  line break, VT signals a new paragraph with associated spacing.
  Payload: None.
  Context: Text section.
  Plain-text extraction: Stripped (replaced by double LF by convention).


--- 0x0C PAGE_BREAK (FF): Page Break ---

  Purpose: Starts a new rendering page or screen. Used in multi-page
  documents or long-form content.
  Payload: None.
  Context: Text section.


--- 0x0D CARRIAGE_RETURN (CR): Carriage Return ---

  Purpose: Standard Windows line break component. PRESERVED in plain-text
  extraction. Typically paired with LF (CR+LF).
  Payload: None.
  Context: Text section.


--- 0x0E LINK_START (SO): Link Start ---

  Purpose: Marks the beginning of a hyperlink. The text between SO and SI
  (0x0F) is the visible link text.

  Payload: [Type: 1 byte] [Length: 1 byte] [Target: N bytes]

  Link types:
    0 = URL (UTF-8 string, Length = byte count of URL)
    1 = QWeb Page ID (fixed-size identifier, Length = size of ID)
    2 = Mailbox Address (7 bytes: coin_group(2) + denom(1) + SN(4))
    3 = Action/Script trigger (1 byte action ID, for Logic section)
    4-255 = Reserved for future types

  Context: Text section.
  Usage:
    [SO] [0x00] [0x16] [https://example.com] Click here [SI]
    (Type=URL, Length=22, Target="https://example.com", Text="Click here")


--- 0x0F LINK_END (SI): Link End ---

  Purpose: Marks the end of a hyperlink started by SO (0x0E).
  Payload: None.
  Context: Text section. Must be preceded by a matching SO.


--- 0x10 DATA_ESCAPE (DLE): Data Escape ---

  Purpose: Bypasses control character interpretation for a block of raw
  binary data. Used when binary content must appear inline in the text
  section (e.g., inline SVG data, binary tokens).

  Payload: [Length: 2 bytes LE] [Raw data: N bytes]
  The parser reads Length bytes without interpreting any control characters
  within the raw data block.

  Context: Text section.
  Note: For Phase II, this is primarily a safety mechanism. Most binary
  data should be in the Resources section.


--- 0x11 STYLE_TEXT (DC1): Apply Text Style ---

  Purpose: Switches the current text formatting to the specified style from
  the Text Styles sub-table in the Styles section.

  Payload: [Index: 1 byte] (0-255, index into Text Styles sub-table)
  The style remains active until another DC1, a DC4 (pop), or ETX.

  Context: Text section.
  Usage:
    Normal text [DC1 0x02] Bold red text [DC1 0x00] Back to default
    Normal text [DC1 0x02] Bold red text [DC4] Back to previous style


--- 0x12 STYLE_CONTAINER (DC2): Apply Container Style ---

  Purpose: Starts a new container (panel, div-equivalent) with the specified
  composite style from the Container Composite Styles sub-table.

  Payload: [Index: 1 byte] (0-255, index into Container Composite sub-table)
  The container holds all content until a matching ETB (0x17) or DC4 (0x14).

  Context: Text section.
  Usage:
    [DC2 0x01] Content inside container styled with composite style #1 [ETB]


--- 0x13 STYLE_TABLE (DC3): Apply Table Style ---

  Purpose: Starts a table with the specified style from the Table Styles
  sub-table. Table cells are separated by US (0x1F). Table rows are
  separated by RS (0x1E) within the text section (note: RS has dual use --
  in Styles section it separates style records; in Text section within a
  table context it separates rows).

  Payload: [Index: 1 byte] (0-255, index into Table Styles sub-table)
  The table continues until ETB (0x17).

  Context: Text section.
  Usage:
    [DC3 0x00] Cell1 [US] Cell2 [US] Cell3 [RS] Cell4 [US] Cell5 [US] Cell6 [ETB]
    (Table style #0, 2 rows x 3 columns)

  DESIGN NOTE: Gemini proposed DC3 for "Apply Layer Style" instead of table
  style. Layer targeting is handled as a PROPERTY of the container style
  record (the composite style includes a layer ID field) rather than a
  separate inline command. This keeps layer management in the styles section
  where it belongs and frees DC3 for the more commonly needed table command.


--- 0x14 STYLE_END (DC4): End Style / Pop ---

  Purpose: Reverts to the previous style on the style stack. Each DC1, DC2,
  or DC3 pushes a style; DC4 pops back to what was active before.

  Payload: None.
  Context: Text section.
  Usage:
    [DC1 0x00] Default text [DC1 0x03] Highlighted text [DC4] Default again


--- 0x15 ELEMENT_ID (NAK): Element ID ---

  Purpose: Assigns a stable, unique ID to the NEXT element in the text
  stream. The Logic section uses these IDs for CRUD operations (create,
  read, update, delete content programmatically).

  Payload: [ID: 1 byte] (0-254; 0xFF is reserved as escape to 2-byte ID)
  If ID = 0xFF: [0xFF] [Extended ID: 2 bytes LE] (for documents needing
  more than 254 addressable elements).

  Context: Text section. Appears immediately before the element it labels.
  IDs MUST be unique within a document and MUST remain stable across
  re-serialization.

  Usage:
    [NAK 0x01] [DC2 0x05] This container has ID=1 [ETB]
    [NAK 0x02] [DC1 0x03] This text span has ID=2 [DC4]


--- 0x16 IMAGE (SYN): Insert Image ---

  Purpose: Inserts an image at the current position in the text flow. The
  image is identified by an Image Definition index (from the Image
  Definitions sub-table in the Styles section). The Image Definition
  record contains the Resource ID, width, height, fit mode, alignment,
  and border style -- everything needed to display the image.

  Payload: [Image Definition Index: 1 byte]
  The index references the Image Definitions sub-table in the Styles
  section (0-based). The Image Definition then points to a Resource ID
  in the Resources section for the actual binary data.

  Indirection chain: SYN [index] -> Image Definition -> Resource ID -> data

  If the resource is not available (section skipped or resource stripped),
  the renderer uses the width/height from the Image Definition to show a
  placeholder or reflow text.

  Context: Text section.


--- 0x17 BLOCK_END (ETB): End of Block ---

  Purpose: Closes the current block-level element (container, table, nav
  bar, or item block). Pairs with DC2, DC3, or EM.

  Payload: None.
  Context: Text section.


--- 0x18 RESERVED_HIDE (CAN): Reserved (Phase III) ---

  Possible future use: "Initially Hidden" -- element is parsed but not
  rendered until the Logic section or user action reveals it. Useful for
  modals (defined in text but hidden by default).
  NOT functional in Phase II.


--- 0x19 ITEM_BLOCK (EM): Start Item Block ---

  Purpose: Begins a structured item block (list, nav bar, or other
  repeating-item element). Items within the block are separated by
  US (0x1F). The block ends with ETB (0x17).

  Payload: [Type: 1 byte] [Style Index: 1 byte]

  Item block types:
    0 = Unordered list (bullets)
    1 = Ordered list (numbered)
    2 = Nav bar (horizontal/vertical navigation)
    3 = Definition list (term/definition pairs, separated by US)
    4-255 = Reserved for future item block types

  Style Index references the appropriate sub-table:
    Types 0-1: List Styles sub-table (not yet defined, use default)
    Type 2: Nav Bar Styles sub-table
    Type 3: List Styles sub-table

  Context: Text section.
  Usage (nav bar):
    [EM 0x02 0x00] [SO ...] Home [SI] [US] [SO ...] About [SI] [US] [SO ...] Contact [SI] [ETB]
    (Nav bar, style #0, three items separated by US)

  Usage (unordered list):
    [EM 0x00 0x00] First item [US] Second item [US] Third item [ETB]


--- 0x1A AI_PROMPT (SUB): AI Prompt ---

  Purpose: Embeds an AI prompt that the RECEIVER's client evaluates to
  generate a style, image, or layout modification. This reduces document
  size by replacing complex style definitions with natural language.

  Payload: [Type: 1 byte] [Length: 2 bytes LE] [UTF-8 prompt text: N bytes]

  AI prompt types:
    0 = Style prompt (AI generates a visual style)
    1 = Image prompt (AI generates an image)
    2 = Layout prompt (AI modifies the page layout)
    3-255 = Reserved

  If the client has no AI capability, it applies the user's configured
  default style/behavior (stored in client settings, customizable).

  Context: Text section.
  Usage:
    [SUB 0x00 0x2C 0x00] Create a cloud-like style with blue sky and
    fluffy whiteness [DC1 0x00] Fallback text continues here...

  Maximum prompt length: 65535 bytes (2-byte LE length field).


--- 0x1B ESCAPE (ESC): Extended Commands (Phase III) ---

  Purpose: Reserved for future extended command sequences that go beyond
  the 32 control characters. When the 32 primary codes are insufficient,
  ESC introduces a secondary command byte, effectively giving 256 additional
  commands.

  Payload: [Command Code: 1 byte] [Payload: variable per command]
  No ESC commands are defined in Phase II. The parser should skip ESC +
  the next byte (and any associated payload, if a length is provided).

  Context: Both structural and text.


--- 0x1C SECTION_SEP (FS): File Separator ---

  Purpose: Separates the major sections of a CBDF document.
  Document structure: [Meta] FS [Styles] FS [Text] FS [Resources] FS [Logic]
  (Text before Resources enables abortable downloads -- clients can stop
  after receiving text without downloading images.)

  Payload: None.
  Context: Structural (document level).
  Always present even if a section is empty (zero-length section = FS
  immediately followed by the section length prefix of 0x00000000).

  Exception: If the meta section contains an EOF flag, NO FS markers follow.


--- 0x1D GROUP_SEP (GS): Group Separator ---

  Purpose: Separates style sub-tables within the Styles section.
  Each GS marks the boundary between different style categories
  (background, border, text, nav bar, etc.).

  Payload: None.
  Context: Structural (Styles section).
  Empty sub-tables: GS immediately followed by the next GS (or FS).


--- 0x1E RECORD_SEP (RS): Record Separator ---

  Purpose: Separates individual records within a style sub-table.
  Also used within the Text section to separate TABLE ROWS when inside
  a DC3 table context.

  Payload: None.
  Context: Structural (Styles section) and Text section (within tables).

  DUAL-USE SAFETY NOTE: There is no ambiguity because the contexts are
  non-overlapping. RS only appears in two places:
    a) Styles section: Between style records within a GS-delimited sub-table.
       The parser is in "styles mode" and knows RS means record boundary.
    b) Text section: ONLY inside a DC3 table block. The parser tracks its
       nesting state (it entered a table via DC3) and knows RS means row
       separator. RS never appears in free-flowing body text.
  A parser that tracks its context (styles vs text, and within text whether
  it's inside a DC3 block) will never misinterpret RS.


--- 0x1F UNIT_SEP (US): Unit Separator ---

  Purpose: Separates items within a structured item block (started by EM
  0x19). Used for nav bar items, list items, and table cells.

  Payload: None.
  Context: Text section (within EM blocks and DC3 tables).

5. STYLE STACK BEHAVIOR
================================================================================

  The text section maintains a STYLE STACK. Commands that push/pop:

    PUSH (new style context):
      STYLE_TEXT [index]      -- pushes text style
      STYLE_CONTAINER [index] -- pushes container style (also opens block)
      STYLE_TABLE [index]     -- pushes table style (also opens block)

    POP (revert to previous):
      STYLE_END               -- pops one level (reverts style, does NOT close block)
      BLOCK_END               -- closes block AND pops associated style

    REPLACE (change without push):
      STYLE_TEXT [index] when already in a STYLE_TEXT context -- replaces
      top of stack (implementation choice: may push or replace; recommend
      push for consistent nesting behavior)

  Example nesting:
    [STYLE_CONTAINER 0x01]  -- push container style #1 (stack: [C1])
      [STYLE_TEXT 0x00]     -- push text style #0 (stack: [C1, T0])
        Normal text
        [STYLE_TEXT 0x03]   -- push text style #3 (stack: [C1, T0, T3])
          Bold text
        [STYLE_END]         -- pop (stack: [C1, T0])
        Normal again
      [STYLE_END]           -- pop (stack: [C1])
    [BLOCK_END]             -- close container, pop (stack: [])


6. CROSS-MODEL ALIGNMENT NOTES
================================================================================

  This table resolves the following disagreements between models:

  a) STYLE_TABLE (0x13): Assigned to TABLE STYLE (cla/gpt proposal), not
     layer targeting (Gemini proposal). Layer IDs are a property of the
     container style record, not an inline command.

  b) AI_PROMPT (0x1A): Uses [Type:1][Len:2 LE][Text:N] format (cla/gpt
     proposal), not [Len:1][Text:N] (Gemini proposal). 2-byte length
     supports prompts up to 65535 bytes.

  c) Phase III reservations: RESERVED_05, RESERVED_06, RESERVED_ALERT,
     RESERVED_BKSP, RESERVED_HIDE, ESCAPE are reserved for future
     interactive features. NOT functional in Phase II.

  d) Nav bar: Uses ITEM_BLOCK (0x19) with type byte, not a dedicated
     control code. This unified "item block" approach covers lists, nav
     bars, and other repeating-item structures with a single control code.

  e) DATA_ESCAPE (0x10): Includes 2-byte LE length prefix so the parser
     knows exactly how many bytes to skip. Without a length, the parser
     cannot resume command interpretation.

7. C ENUMERATION
================================================================================

The following enum can be used directly in C parser/serializer code. It maps
human-readable CBDF command names to their byte values, so developers never
need to remember hex codes or ASCII symbols.

  /**
   * cbdf_cmd_t - CBDF control character command codes.
   *
   * These are the 32 ASCII control characters (0x00-0x1F) repurposed
   * as CBDF inline commands and structural separators.
   *
   * Usage: if (byte == STYLE_TEXT) { index = next_byte(); ... }
   */
  typedef enum {
      /* -- Padding & Terminators -- */
      NOP              = 0x00,  /* No operation / alignment padding      */
      SUBJECT_START    = 0x01,  /* Styled subject begins (inside STX)    */
      TEXT_START        = 0x02,  /* Start of text body                    */
      TEXT_END          = 0x03,  /* End of text body (validation marker)  */
      DOC_END           = 0x04,  /* End of document                       */

      /* -- Reserved for Phase III -- */
      RESERVED_05       = 0x05,  /* Future: interactive resource request  */
      RESERVED_06       = 0x06,  /* Future: interactive acknowledgment    */
      RESERVED_ALERT    = 0x07,  /* Future: client alert / sound          */
      RESERVED_BKSP     = 0x08,  /* Future: typing / animation effect     */

      /* -- Whitespace (preserved in plain-text extraction) -- */
      TAB               = 0x09,  /* Horizontal tab                        */
      LINE_BREAK        = 0x0A,  /* Line feed (Unix line break)           */
      PARA_BREAK        = 0x0B,  /* Paragraph break (extra spacing)       */
      PAGE_BREAK        = 0x0C,  /* Page break / new page                 */
      CARRIAGE_RETURN   = 0x0D,  /* Carriage return (Windows line break)  */

      /* -- Links -- */
      LINK_START        = 0x0E,  /* Hyperlink start: [Type][Len][Target]  */
      LINK_END          = 0x0F,  /* Hyperlink end                         */

      /* -- Data -- */
      DATA_ESCAPE       = 0x10,  /* Raw binary data: [Len:2 LE][Data]     */

      /* -- Style Application -- */
      STYLE_TEXT        = 0x11,  /* Apply text style: [Index]             */
      STYLE_CONTAINER   = 0x12,  /* Apply container style: [Index]        */
      STYLE_TABLE       = 0x13,  /* Apply table style: [Index]            */
      STYLE_END         = 0x14,  /* Pop style stack / revert              */

      /* -- Elements -- */
      ELEMENT_ID        = 0x15,  /* Assign stable ID: [ID]                */
      IMAGE             = 0x16,  /* Insert image: [ImageDefIndex]         */
      BLOCK_END         = 0x17,  /* Close current block element           */

      /* -- Reserved for Phase III -- */
      RESERVED_HIDE     = 0x18,  /* Future: initially hidden / toggle     */

      /* -- Structured Content -- */
      ITEM_BLOCK        = 0x19,  /* List/nav/etc: [Type][StyleIndex]      */
      AI_PROMPT         = 0x1A,  /* AI prompt: [Type][Len:2 LE][UTF-8]   */

      /* -- Reserved for Phase III -- */
      ESCAPE            = 0x1B,  /* Extended commands: [Code][Payload]    */

      /* -- Structural Separators -- */
      SECTION_SEP       = 0x1C,  /* Between major sections (FS)           */
      GROUP_SEP         = 0x1D,  /* Between style sub-tables (GS)         */
      RECORD_SEP        = 0x1E,  /* Between records / table rows (RS)     */
      UNIT_SEP          = 0x1F   /* Between items in a list/nav (US)      */
  } cbdf_cmd_t;


  /* ----- Link types (payload of LINK_START) ----- */
  typedef enum {
      LINK_URL          = 0x00,  /* URL string                            */
      LINK_QWEB_PAGE    = 0x01,  /* QWeb page ID                          */
      LINK_MAILBOX      = 0x02,  /* Mailbox address (7 bytes)             */
      LINK_ACTION       = 0x03   /* Logic section action trigger           */
  } cbdf_link_type_t;


  /* ----- Item block types (payload of ITEM_BLOCK) ----- */
  typedef enum {
      ITEM_UNORDERED    = 0x00,  /* Unordered list (bullets)              */
      ITEM_ORDERED      = 0x01,  /* Ordered list (numbered)               */
      ITEM_NAV          = 0x02,  /* Navigation bar                        */
      ITEM_DEFINITION   = 0x03   /* Definition list (term/value pairs)    */
  } cbdf_item_type_t;


  /* ----- AI prompt types (payload of AI_PROMPT) ----- */
  typedef enum {
      AI_STYLE          = 0x00,  /* Generate a visual style               */
